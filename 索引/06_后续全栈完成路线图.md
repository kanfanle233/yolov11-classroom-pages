## 推荐执行顺序

1. 冻结一个主线 case：`front_002_A8_adaptive_sliced_artifact_deblur_opencv`。
2. 生成并人工填写 `front_002_rear_gt.jsonl`。
3. 用 `19_eval_rear_row_metrics.py` 跑正式指标，确认指标不是空。
4. 重新跑 `16_run_rear_row_sr_ablation.py --run_pipeline 0 --gt_jsonl ...` 汇总消融表。
5. 如果正式指标显示 A8 召回提升但 IDSW 上升，先优化 tracking/seat prior，不继续盲目做 SR。
6. 如果正式指标显示检测漏检仍是主因，再做更强后排增强或模型微调。
7. 后端指标稳定后，开始前端 D3 数据 manifest 和页面实现。
8. 前端完成后，跑一次端到端 demo，确保图表能链接到视频、帧号、学生 ID 和产物文件。

## 阻塞点

| 阻塞点 | 影响 | 解决方式 |
| --- | --- | --- |
| 缺真实 GT | 不能写正式准确率 | 先标 front_002 30-60 秒后排 ROI |
| 后排小目标仍漏检 | recall/PCK 低 | ROI sliced、dense/adaptive、SR/deblur、必要时微调 |
| ID switch | IDF1/HOTA 低 | seat prior、ReID、跟踪参数、pose-backbone 固化 |
| 行为 unmatched 多 | timeline 缺行为 | 行为框扩展匹配、Hungarian matching、行为模型微调 |
| 前端数据不统一 | 页面难维护 | 先做 `frontend_data_manifest.json` 和 schema 文档 |
| 外部 SR 后端缺失 | A4/A5/A6/A10/A11 无法跑 | 作为可选消融，先不阻塞主线 |

## 完成定义

项目可以宣布“当前阶段完成”的最低条件：

- 至少 `front_002` 有人工 GT，正式指标表非空。
- A0/A1/A2/A3/A8 至少 5 个变体有完整 pipeline 和指标。
- 主线 demo 视频可播放，显示骨架、学生 ID、8 类行为、未链接行为框 debug。
- `pipeline_contract_v2_report.json status=ok`。
- `actions.fusion_v2.jsonl` 中 `track_id>=200000` 为 0。
- 前端可以展示视频、学生时间线、指标对比、后排增强对比和失败案例。
