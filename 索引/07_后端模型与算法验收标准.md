# 07 后端模型与算法验收标准

更新时间：2026-05-04

## 验收原则

- 不只看视频观感。
- 不只看单个 mAP。
- 不把 proxy 指标当 formal 指标。
- 不把已有脚本或接口自动等同于已闭环能力。

| 维度 | 核心证据 | 当前状态 |

## 当前对模型与算法的真实判断

### 已落地现状

- 行为检测模型已有独立 test 指标。
- 后排切片/SR 消融已经形成稳定实验矩阵。
- `front_046` formal GT 已经能支撑 person/tracking/behavior 基础结论。
- `Pipeline Contract` 已经是系统级硬约束，不只是分析脚本。

### 已有代码或结果，但证据仍不足

- 音频上下文增强与 LLM 语义融合有脚本、有样本，但样本量还不足以做强 benchmark 主结论。
- `index_v2.html` 和 BFF 已存在，但展示层还属于“已具备能力，仍需整合”的阶段。
- `front_001/front_002` 的后排增强主要仍是 proxy 口径。

### 当前不应验收通过的方向

- OCR / 敏感文本识别。
- “音视频双重验证已充分成立”。
- “所有 case 都有 formal GT”。
- “底层 YOLO 增强模块已完成训练闭环”。

## 当前最合理的通过标准

### 工程层

- 主流程能稳定输出 pose、track、behavior、fusion、timeline、contract 产物。
- bundle 可被 API 和前端页面直接读取。
- BFF endpoint 能返回统一 view-model。

### 论文层

- 检测主结果可写。
- `front_046` formal 可写。
- `front_001/front_002` 只能写 proxy 或工程证据。
- 音频与 LLM 部分当前应写成上下文增强与系统分析，而不是强 benchmark。

### 展示层

- `paper_demo.html` 能完成单 case 展示。
- `index_v2.html` 已具备 BFF 接入基础。
- `front_046_A8` bundle 已能支撑 formal + bundle + API 的联合展示。

## 当前不建议作为“完成依据”的信号

- 只看某个 contact sheet 更清楚。
- 只看 `tracked_students` 增加。
- 只看 smoke test。
- 只看 `front_001/front_002` 的 proxy 提升。
- 只看“接口存在”，不看接口是否有对应真实样本与产物。
