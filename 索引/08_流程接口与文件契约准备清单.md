### BFF API

已落地：

- `/api/v1/visualization/case_data`

当前定位：

- Bundle API：原子数据读取层。
- BFF API：给 `index_v2.html` 一类页面使用的聚合视图层。

## 当前必须保持的硬约束

| 契约 | 当前要求 |
| --- | --- |
| Hybrid 身份主干 | `student_track_students == tracked_students` |
| Fusion 合法性 | `actions.fusion_v2.jsonl` 不允许 `track_id >= 200000` |
| Formal 指标口径 | `gt_status=ok` 才能写 formal 结论 |
| Bundle 读写 | 前端通过 manifest 与 API 读数据，不直接硬编码原始 case 路径 |
| 前端状态分层 | 必须区分 formal / proxy / missing |

## 当前最容易出错的地方

| 问题 | 当前处理原则 |
| --- | --- |
| 把 `front_001/front_002` proxy 当 formal | 明确禁止 |
| 把 `front_046` 的 GT 用到别的视频 | 明确禁止 |
| 把 `index_v2.html` 方案能力写成已全面上线 | 只能写“已有代码与接口” |
| 把 `verified_events` 中的语义能力写成强 benchmark | 目前只能保守写 |

## 当前建议的接口说明写法

- `paper_demo.html` 走 bundle API 足够。
- `index_v2.html` 走 BFF API 更合适。
- 无论走哪条链，bundle manifest 都应继续保留为统一入口。
