# 09 前端 D3 可视化实施方案

更新时间：2026-05-04

## 当前前端事实基线

| 组件 | 当前状态 | 说明 |
| --- | --- | --- |
| `paper_demo.html` | 已落地 | 单 case 论文演示页 |
| `index.html` | 已存在 | 旧版/大页模板 |
| `index_v2.html` | 已存在 | BFF 驱动模板，已直接引用 `/api/v1/visualization/case_data` |
| Bundle API | 已落地 | `/api/bundle/*` |
| BFF API | 已落地 | `/api/v1/visualization/case_data` |
| Bundle 数据 | 已落地 | 当前至少 6 个 bundle case |

## 当前已实现能力

- 视频播放与预览图资产。
- 学生行为时间线。
- bundle manifest 驱动的数据装配。
- `front_046_A8` formal case 的 bundle 与 API 读取。
- `paper_demo.html` 的论文展示路径。
- `index_v2.html` 对 BFF 的直接接线。

## 当前应写成“已有代码/接口，但未完全闭环”的能力

- `index_v2.html` 是否成为默认主展示页。
- 基于 BFF 的完整多面板交互。
- failure cases 的丰富内容与联动展示。
- 音频上下文证据在前端的系统性呈现。
- 不确定样本、冲突样本、人工审核流。

## 当前不应写成已完成的能力

- OCR 文本流展示。
- 音视频双重验证的完整可视化闭环。
- 全量多 case formal 对比平台。
- 最终产品化信息架构。

## 当前推荐页面分工

| 页面 | 定位 | 数据来源 |
| --- | --- | --- |
| `paper_demo.html` | 稳定单 case 论文页 | bundle API |
| `index_v2.html` | BFF 驱动的下一代展示页 | BFF API + bundle list |
| `index.html` | 旧版大页或保留页 | 旧交互逻辑 |

## 当前推荐数据分层

| 层 | 用途 |
| --- | --- |
| `frontend_data_manifest.json` | 统一入口 |
| `timeline_students.json` / `tracks_sampled.json` | 交互主数据 |
| `metrics_summary.json` / `ablation_summary.json` | 指标层 |
| `verified_events.json` | 证据层 |
| `/api/v1/visualization/case_data` | 聚合后的 BFF 视图层 |

## 当前推荐的图表优先级

### 已可以稳定支持

- 学生行为时间线
- SR 消融柱状图
- 指标摘要卡片
- 视频与 contact sheet 联动

### 已有数据基础，可继续增强

- 基于 `verified_events` 的证据面板
- 基于 BFF 的 case 切换
- 多 case 对比视图
- failure cases 浏览

### 仍属规划

- OCR 文本叠加
- 大规模 uncertain/mismatch 审核流
- 更强的音轨与语义对齐可视化

## 当前最稳妥的实施原则

- 先保 `paper_demo.html` 的可复现展示。
- `index_v2.html` 明确写成“已接 BFF 的新模板”，不要直接写成已经替代旧页。
- 前端指标页必须区分 formal / proxy / missing。
- bundle 与 BFF 的存在是事实；样本闭环与展示完备度要单独说明。

## 当前演示建议

1. 用 `front_046_A8` 做 formal + bundle + API 演示。
2. 用 `front_002_A8` 做“proxy case”对照。
3. 用 `front_45618_sliced` 或 `front_1885_sliced` 做音频上下文补充演示，但表述保持保守。
